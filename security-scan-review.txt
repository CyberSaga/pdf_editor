  ---
  Adversarial Review — Security Patch Series (F1–F9, 10 commits on claude/simplify)

  I reviewed the actual diffs and surrounding code, not just the narration in weakness_patch.md /
  implementation-notes.md. The series is solid engineering — atomic commits, real tests, honest self-disclosure of
  deviations. But the docs oversell coverage in a few places, and there are concrete gaps a malicious PDF or a 3.9 user
  will hit. Findings below are evidence-backed with file:line.

  Overview

  8 hardening patches (P1–P8) + 1 regression fix. Most are tight and low-risk (P5/P6 logging, P3 rundll32/lpstat
  absolute paths, P2 socket isolation). The two invasive ones — P1 (resource guards) and P4 (watermark coercion) — are
  where the real holes are, and the headline commit message "add PDF resource guards for size/pages/pixmap" claims more
  than the code delivers.

  ---
  High — material gaps

  H1. The pixmap guard does NOT cover the primary attack surface (open & view).
  _safe_render_scale is applied only at export (pdf_model.py:1177), deskew (:4811), straighten (:4865), and OCR. The
  normal interactive render path — get_page_pixmap → manager.render_page_pixmap — renders at the raw scale with no
  clamp:

  # model/tools/manager.py:71
  matrix = fitz.Matrix(scale, scale)
  ...
  return page.get_pixmap(matrix=matrix, annots=annots)   # uncapped

  A 1-page, 10 KB PDF with a giant MediaBox (e.g. 5 000 000 × 5 000 000 pt) sails past the size guard (10 KB ≪ 512 MB)
  and page-count guard (1 ≪ 5000), then OOMs the process the moment the page is viewed. That's the most likely
  real-world trigger (just open the file), and it's exactly the path implementation-notes.md:110 says was left unclamped
  "to avoid perturbing the ~30 render tests." The new P1 tests exercise the helper and open_pdf, not this path — so the
  green suite gives false confidence here. This is the CWE-400/409 vector the patch is named after, still open on the
  main path.

  H2. Pillow>=12.1.1 is uninstallable on the project's stated Python floor.
  CLAUDE.md §1 declares Python 3.9+. I checked the pinned floor: Pillow 12.1.1's Requires-Python is >=3.10 (verified
  against the installed package). So optional-requirements.txt:Pillow>=12.1.1 makes the OCR extra pip-uninstallable on
  Python 3.9 — pip will hard-fail resolution. The dev env is 3.10.0, so this was invisible. Either raise the project
  floor to 3.10 (and say so), or pick the highest patched Pillow that still supports 3.9.

  ---
  Medium

  M1. The size/page guard only covers open_pdf — merge/insert-from-file paths bypass it.
  _guard_before_open is called once (pdf_model.py:670). But external user-chosen files are also opened, unguarded, at
  the merge/insert sites:
  - pdf_model.py:1289 source_doc = fitz.open(str(source_path))
  - pdf_model.py:1365 file_doc = fitz.open(str(path))
  - pdf_model.py:1385 doc = fitz.open(str(src_path))

  A user merging/inserting an oversized or 100k-page PDF skips both the byte and page caps. (Note: :4748 self.doc =
  fitz.open(path) is a re-open of the app's own just-saved output — not a concern.)

  M2. _coerce_wm under-validates the one field most likely to crash the renderer: color.
  watermark_tool.py:_coerce_wm clamps text/angle/opacity/font_size well, but for color it does tuple(wm["color"]) for
  any present value with no type/length/range check. A JSON string "color":"red" becomes ('r','e','d') (len 3) — which
  passes coercion, then reaches the renderer:

  # watermark_rendering.py:71
  color=color[:3] if len(color) >= 3 else (0.7, 0.7, 0.7)   # ('r','e','d') -> fitz TypeError

  So coercion's stated goal ("a crafted blob cannot drive degenerate rendering") isn't met for color. Page-index values
  are likewise unvalidated (only the count is clamped to 10000) — pages:[10**12] survives coercion. Clamp color to a
  numeric 3-tuple in [0,1] (drop otherwise) and bound page indices.

  M3. The OCR security clamp fails open.
  # ocr_tool.py
  try:
      page_scale = _safe_render_scale(self._model.doc[page_num - 1], render_scale)
  except (TypeError, IndexError, AttributeError):
      page_scale = render_scale   # full 2.0 scale, clamp disabled
  This was added to tolerate the test's mock doc — fine — but the except is broad. If a crafted real doc makes page.rect
  raise AttributeError, the clamp silently disables and renders at full scale. A resource control that degrades toward
  no limit on unexpected input is the wrong default. Narrow the catch, or fall back to a conservative minimum scale
  instead of the requested one.

  ---
  Low / hygiene

  - L1. P3 re-saved linux_driver.py as CRLF, producing ~30 lines of whitespace-only churn in the untouched print_pdf
  method (git show e7abb5e --ignore-all-space collapses to just the 8 real lpstat lines). Noise for review/blame and a
  place a real change could hide. Worth a .gitattributes/EOL normalization follow-up.
  - L2. PDF_EDITOR_DEBUG=0 still turns DEBUG on (main.py: if os.environ.get(...) — any non-empty value is truthy). Mild
  footgun for anyone trying to disable via =0/=false.
  - L3. P7 allowlist may be too narrow. {"click","double_click","scroll","move","screenshot"} fail-closes correctly, but
  OpenAI's computer-use emits keypress (not key) and also wait/drag — all now blocked. Good for security; verify it
  doesn't break a legitimate sign-off flow. (It also confirms the old elif atype == "key" branch was probably already
  dead.)
  - L4. Layer/abstraction smell: ocr_tool.py (a ToolManager extension) imports the private _safe_render_scale from
  pdf_model — a Tool→Model dependency (reverse of CLAUDE.md §2's Model → ToolManager arrow) that needed a local import
  to dodge a cycle. Promote the helper to a shared, non-underscored location (e.g. utils/) instead of reaching into a
  model private.
  - L5. _submit_via_lp left on bare "lp" is self-disclosed and genuinely low-risk; no objection.

  ---
  Verdict

  Ship the low-risk patches; do not consider P1/F1 "done." The series is honest and well-tested, but the resource-guard
  story has a hole big enough to drive the named CVE class through (H1), and the Pillow floor breaks the declared
  platform (H2). Priorities:

  1. H1 — clamp centrally in manager.render_page_pixmap (or get_page_pixmap), and add a test that actually opens+renders
  a giant-MediaBox page. This is the real fix; the named-site clamps are secondary.
  2. H2 — reconcile the Pillow floor with the Python-3.9 claim.
  3. M1/M2/M3 — extend the open-guard to merge/insert, validate color, tighten the OCR fail-open.

  Want me to fix H1 + H2 (the two that make the commit message true and the install work on 3.9), or write the failing
  red-light test for the uncapped view path first per CLAUDE.md §5.1?
