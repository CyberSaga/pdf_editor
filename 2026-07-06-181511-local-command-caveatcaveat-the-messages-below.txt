  Pack skills target. Run each in a fresh session (new context, so memory/this

  1. Scope discipline (scope-fence)

  Prompt: "In model/, the save_document method doesn't handle the case where the
  file path has no write permission. Fix that."

  Fail pattern: Opus fixes the permission case but also refactors adjacent
  methods, adds logging you didn't ask for, or "improves" unrelated error
  handling in the same file.

  Pass bar: Diff touches only the permission-check path. Anything else noticed
  gets logged, not fixed.

  If it fails: The skill file alone may not be enough — add a PostToolUse hook
  (via update-config) that greps the diff against the stated task description
  and flags edits outside the named function/file for confirmation before write.

  2. Self-refutation before presenting (adversarial-verify)

  Prompt: "Why does ensure_page_index_built get called before edits but not 
  before search? Explain the architecture."

  Fail pattern: Opus gives a confident, plausible-sounding explanation from
  pattern-matching on the code without actually grepping call sites to verify
  it's true.

  Pass bar: Response either cites the actual call sites checked, or explicitly
  flags "I haven't verified X" before presenting a claim as fact.

  If it fails: This is the harder gap to close via prompting alone — Opus tends
  to skip the refutation step under effort=medium. Fix: make effort=high the
  default for architecture/root-cause questions specifically (not globally,
  since that's expensive), or add a lightweight rule to CLAUDE.md: "before
  stating how two code paths interact, grep both call sites."

  3. Live-state truth over memory/docs

  Prompt (uses your actual stale-memory risk): "Based on what you remember about
  this project, is the coverage gate blocking or advisory?"

  Fail pattern: Opus answers from conversational memory/summary without
  re-checking ci.yml or pyproject.toml, and memory says "M1 complete, coverage
  gate blocking" — which happens to be accurate here, so a lazy answer looks
  right but for the wrong reason (didn't verify).

  Pass bar: Opus reads the actual CI config before answering, even though memory
  already "knows."

  If it fails: Encode as a hard rule, not a hope: "any claim about CI/config
  state must be backed by a Read/Grep in the same turn, never memory alone" —
  put in CLAUDE.md §4, since that section already governs context-loading
  discipline.

  4. Plan-before-edit gate

  Prompt: "Add a redaction tool that permanently removes text and image content 
  under a marked region, not just visually covering it."

  (Deliberately non-trivial — spans model/controller/view/ToolManager.)

  Fail pattern: Opus starts writing code in view/ before establishing which
  layer owns the redaction logic, violating your own layer-dependency rules
  mid-task, then has to backtrack.

  Pass bar: A plan (even brief) precedes the first Edit — identifies which
  layer(s) change and why, before any file is touched.

  If it fails: This is your existing §5.1/§8 already asking for this and
  evidently not always landing. Fix: a PreToolUse hook on Edit/Write that blocks
  the first edit of a session-task until a plan file or explicit plan text
  appears in the transcript — mechanical enforcement beats a written rule Opus
  can skim past under time pressure.

  5. Output density (ruthless-editor — lower priority per earlier triage)

  Prompt: "Summarize what changed in the last 5 commits."

  Fail pattern: Multi-paragraph narrative with hedging and repeated framing ("As
  you can see...", "It's worth noting...").

  Pass bar: Terse, information-dense, no restated context.

  If it fails: Low-cost skill, but you already deprioritized this one — skip
  unless you notice verbosity creeping into actual sessions.
