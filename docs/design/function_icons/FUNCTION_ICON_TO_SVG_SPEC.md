# Function Icon Design Specification

## Purpose

This document defines the design rules for the `01` to `32` toolbar function icons.
These icons are small-size toolbar assets, so clarity at 16-24 px is the primary goal.

The icon set should read as a consistent hamster-themed function icon system, not as
miniature illustrations copied from large PNG cards.

## Format

- Primary format: SVG
- Canvas: `viewBox="0 0 24 24"`
- Intended display size: 18 px, 20 px, or 24 px
- Source icons should remain editable SVG files.
- PNG may be exported only as a fallback for toolkits that cannot render SVG reliably.

## Design Priorities

1. Function recognition
2. Clean silhouette
3. Consistent hamster identity
4. Consistent toolbar appearance
5. Decorative detail

At toolbar size, decorative elements should be removed when they compete with function
recognition.

## Visual System

### Stroke

- Main stroke color: `#111827`
- Main stroke width: `1.6` to `1.8`
- Detail stroke width: `0.8` to `1.2`
- Stroke caps: `round`
- Stroke joins: `round`

Avoid thin strokes below `0.8`, because they become weak or disappear at small sizes.

### Color Palette

- Primary outline: `#111827`
- Hamster fill: `#FFE3B0`
- Inner ear / cheek accent: `#F6B7C4` or `#FF9FB3`
- Function teal: `#67D6CB`
- Function blue-gray: `#5B6F95`
- White highlight: `#FFFFFF`
- Disabled state can be generated by UI code, not by separate source assets.

Use a limited palette. Each icon should usually contain:

- one outline color
- one hamster color
- one function color
- optional white highlight

## Hamster Motif

The hamster should be recognizable but simplified.

Recommended small-size features:

- round ears
- cream/yellow face
- two black eyes
- tiny nose or short mouth line
- optional cheek dots only when space allows

Avoid:

- full body poses unless the function shape leaves enough room
- complex paws, fur marks, shadows, or facial expressions
- making the hamster larger than the functional symbol

The hamster is a brand cue. The function symbol must remain the dominant readable shape.

## Composition Rules

Each icon should use two layers:

1. Function layer: the tool/action symbol, such as cursor, text, scanner, eraser, crop,
   hand, selection box, page, magnifier, or annotation mark.
2. Hamster layer: a simplified hamster head, ears, or small mascot accent.

The function layer should be readable even if the hamster details are mentally ignored.

Do not include the original large-card elements in toolbar icons:

- rounded white card background
- card border
- drop shadow
- bottom ground line
- large illustration-style lighting

These details waste pixels and reduce toolbar clarity.

## Transparency

Semi-transparent effects are allowed, but only in simple forms.

Recommended:

- one translucent beam, selection area, or highlight
- opacity between `0.18` and `0.46`

Avoid:

- layered shadows
- blurred glows
- multiple transparent gradients
- transparency that overlaps tiny facial details

## OCR / Text Scan Icon Example Rules

For the OCR/text-scan icon:

- Use a scanner bar at the top.
- Use a single translucent teal beam.
- Use four corner brackets to show recognition/selection.
- Use a large `T` to make the text function unmistakable.
- Place a simplified hamster face inside or near the scan area.

The `T` and scanner must remain clear at 20 px.

## Export Checklist

Before accepting each icon:

- It uses `viewBox="0 0 24 24"`.
- It remains readable at 20 px.
- It still reads acceptably at 16 px.
- The main function is recognizable without reading the filename.
- The hamster cue is present but does not obscure the function.
- Stroke widths are consistent with the rest of the set.
- There is no card background or decorative shadow from the original PNG.
- SVG paths are reasonably clean and editable.

## Naming

Use stable names that describe the function, not only the source number.

Recommended pattern:

```text
01_text_scan.svg
02_select_cursor.svg
03_...
```

If the UI already depends on numeric names, keep the numeric prefix and add a semantic
suffix.

## Implementation Recommendation

The application should load SVG first and fall back to PNG only when an SVG is missing
or unsupported.

Recommended asset layout:

```text
function_icons/
  svg/
    01_text_scan.svg
    02_select_cursor.svg
  png/
    01_text_scan_24.png
    02_select_cursor_24.png
```

The SVG files should be treated as source assets. PNG files should be generated exports.
