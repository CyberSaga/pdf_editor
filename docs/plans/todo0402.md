# Deferred Tasks — 2026-04-02

Tasks identified during Week 2 CEO + Eng + Design review that are deferred to Month 3+.

## Text Selection Highlight

- What: Show visible selection highlight (e.g., blue rect overlay) when user clicks/hovers text blocks before entering edit mode.
- Why: Acrobat highlights the target text block before editing begins. Currently our editor goes straight from click to editor open with no intermediate visual selection state.
- Effort: M (requires QGraphicsScene overlay or rect-mode implementation)
- Priority: P3
- Risk of deferring: Low — users click to edit and block outlines provide discoverability. Selection is nice-to-have polish.
- Depends on / blocked by: None.

## Inline Formatting Preservation (Bold/Italic/Underline)

- What: Preserve and allow editing of rich text formatting (bold, italic, underline, mixed font weights) within the text editor.
- Why: Current editor is plain-text only. Editing styled text strips all formatting. Acrobat preserves font weight, italics, etc.
- Effort: L (requires rich-text edit pipeline rewrite — QTextEdit supports rich text natively, but the model layer's insert_htmlbox pipeline needs per-run style tracking)
- Priority: P2
- Risk of deferring: Medium — users editing styled PDFs lose formatting on every edit. This is the largest remaining fidelity gap vs Acrobat.
- Depends on / blocked by: edit_text() phase extraction (Step 2) should land first — style-per-run logic is much easier to add to `_apply_redact_insert()` than to the monolithic method.

## Live Drag Preview / Ghost Outline

- What: Show a translucent "ghost" outline of the text block at the drag destination during cross-page drag, so users can preview where text will land before releasing.
- Why: Current behavior is "commit then move" — user drags, editor closes, text reappears at new position. Acrobat shows real-time reflow preview as you drag.
- Effort: M-L (complex cross-page rendering; need to track drag position across QGraphicsScene page boundaries)
- Priority: P3
- Risk of deferring: Low — current commit-then-move works correctly, just not as smooth.
- Depends on / blocked by: None, but benefits from TextEditSession consolidation.

## TextEditSession State Consolidation

- What: Replace the 25 scattered view attributes (editing_font_name, editing_color, _editing_page_idx, etc.) with a single `TextEditSession` dataclass created at editor-open time (not finalize time).
- Why: Current state management is fragile — if any cleanup is forgotten in finalize, orphaned attributes linger. A session object provides a single source of truth and enables reactive state updates (e.g., live drag preview, undo state tracking).
- Effort: M (architectural refactor touching pdf_view.py and text_editing.py; no user-visible change)
- Priority: P2
- Risk of deferring: Low short-term, but increases fragility risk over time as more features are added to the editing pipeline.
- Depends on / blocked by: Should be done alongside or after edit_text() phase extraction to avoid two large refactors in flight.

## Viewport 180ms Double-Restore Investigation

- What: Investigate and fix the root cause of viewport instability after text edits that requires a 180ms QTimer double-restore hack (pdf_controller.py:1583-1584).
- Why: The workaround works but suggests an underlying race condition between page reflow and viewport anchoring. If the timing changes (e.g., larger PDFs, slower machines), the hack may break.
- Effort: S-M (investigation may be quick if root cause is obvious; fix depends on what's found)
- Priority: P3
- Risk of deferring: Low — workaround is stable in current testing. Revisit during Month 3 refactor window.
- Depends on / blocked by: None, but easier to investigate after edit_text() phase extraction separates the reflow logic.
