# F2 Child Plan -- Surya OCR Integration

## Goal

Add OCR using Surya without destabilizing the current editing/printing pipeline or forcing heavyweight dependencies into the default app startup path.

## Success Criteria

- OCR can be invoked on selected pages from the app.
- Surya dependencies are optional and isolated behind a clear availability check.
- OCR results are inserted back into the document through typed model/controller flows.
- The UI stays responsive while OCR is running.

## Scope

1. Define an OCR service boundary separate from the PDF model.
2. Add dependency detection and actionable error messaging when Surya is unavailable.
3. Run OCR in a worker/off-main-thread path.
4. Normalize OCR output into page text overlays or insertable text requests.
5. Add progress/error reporting in the view.

## Out of Scope

- Bundling Surya binaries into the repo.
- Full layout-perfect reconstruction in the first tranche.
- Multi-provider OCR abstraction beyond a minimal service boundary.

## Workstreams

- Service layer for OCR execution and result normalization.
- Controller orchestration for page selection, progress, and commit.
- View flow for starting OCR and showing progress/completion.
- Tests for dependency checks, threading boundaries, and result application.

## Risks

- Surya installation complexity on Windows may block adoption.
- OCR latency and memory usage may be high on large documents.
- Low-confidence OCR output can create misleading editable text if inserted blindly.

## Exit Evidence

- Dependency/isolation tests around the OCR service boundary.
- Controller tests proving OCR work does not block the UI thread.
- Manual verification on scanned PDFs with mixed languages/layouts.
