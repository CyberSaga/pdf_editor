&#x20;Recommended Week 2 Sprint (Priority Order)



&#x20;Step 1: Fix size: int -> size: float (Quick Win, \~5 min)



&#x20;- Files: view/text\_editing.py (EditTextRequest + MoveTextRequest dataclasses)

&#x20;- What: Change size: int to size: float in both dataclasses

&#x20;- Why: PyMuPDF returns float font sizes. Silent truncation (9.5pt -> 9pt) corrupts style fidelity on every edit with

&#x20;fractional sizes

&#x20;- Approach: Update dataclass fields, grep for any callers passing explicit int literals, update test fixtures if

&#x20;needed

&#x20;- Risk: Minimal - Python int is assignment-compatible with float



&#x20;Step 2: edit\_text() Phase Extraction (Medium, \~30 min CC)



&#x20;- File: model/pdf\_model.py (\~line 2323+, \~400 LOC)

&#x20;- What: Decompose edit\_text() into \_resolve\_target(), \_apply\_redact\_insert(), \_verify\_rebuild()

&#x20;- Why: Most battle-tested but hardest-to-debug code. Phase helpers make each step independently testable. This

&#x20;unblocks future UX improvements that touch the edit pipeline.

&#x20;- Approach: Structural extraction only - no behavioral changes. The 7 phases are already documented in the docstring.

&#x20;Extract sequentially, run full test suite after each extraction.

&#x20;- Prerequisite: MoveTextRequest + foundation tests (now landed)

&#x20;- Risk: Moderate - regression risk on core code path. Mitigated by 73 existing tests.



&#x20;Step 3: Error Toasts for Invalid Operations (Medium, \~20 min)



&#x20;- Files: controller/pdf\_controller.py, view/pdf\_view.py

&#x20;- What: Surface clear error feedback when edit operations fail (e.g., editing a non-editable block, font not found,

&#x20;edit on protected page)

&#x20;- Why: Acrobat shows inline error messages. Currently this editor silently fails, leaving users confused.

&#x20;- Approach: Add toast notifications (reuse the mode-switch toast pattern already in pdf\_view.py) at controller

&#x20;validation boundaries. Identify the 3-4 most common failure modes from the edit pipeline.



&#x20;Step 4: Undo/Redo State Feedback (Low effort, High polish)



&#x20;- Files: view/pdf\_view.py, controller/pdf\_controller.py

&#x20;- What: Visual indication of undo/redo availability (e.g., grayed-out toolbar buttons, status bar text)

&#x20;- Why: Users can't tell if undo is available or if redo stack is empty. Acrobat grays out Edit menu items.

&#x20;- Approach: Connect to CommandManager's stack state to enable/disable UI elements after each operation.
