# Plan: Python-native codegraph indexer for pdf_editor

## Context
Referencing codegraph's technique (SQLite graph + FTS5 + query CLI), build an equivalent Python-native indexer using only Python stdlib (`ast`, `sqlite3`). The output is a queryable `.codegraph/graph.db` plus a pre-built `CODEINDEX.md` static index — so future agents either call the query CLI (targeted lookup) or read the index file (full context in one shot).

No npm, no external installs. `ast` and `sqlite3` are both stdlib.

---

## What codegraph does (the method we reference)
- **Parse** source files with a tree-sitter grammar → extract symbol nodes (file, class, function, method) and edges (imports, defines, calls, inherits)
- **Store** in SQLite with an FTS5 virtual table on symbol names/signatures/docstrings
- **Serve** query tools: search (FTS), context (node + neighbors), callers (reverse edges), callees (forward edges), explore (BFS traversal)
- **Static artifact**: the DB persists so agents query it cold without re-parsing

We do the same in Python with `ast` for parsing.

## SQLite schema (graph.db)

```sql
CREATE TABLE nodes (
  id        TEXT PRIMARY KEY,   -- "<file>::<qualname>" e.g. "model/pdf_model.py::PDFModel"
  kind      TEXT NOT NULL,      -- 'file' | 'class' | 'function' | 'method'
  name      TEXT NOT NULL,      -- short name
  qualname  TEXT NOT NULL,      -- dotted qualified name inside file
  file      TEXT NOT NULL,      -- relative path from project root
  line      INTEGER,
  docstring TEXT,
  signature TEXT,               -- def foo(a, b) -> int
  bases     TEXT                -- comma-separated base class names (classes only)
);

CREATE VIRTUAL TABLE nodes_fts USING fts5(
  id UNINDEXED, name, qualname, docstring, signature,
  content=nodes, content_rowid=rowid
);

CREATE TABLE edges (
  src   TEXT NOT NULL,
  dst   TEXT NOT NULL,
  kind  TEXT NOT NULL,          -- 'defines' | 'imports' | 'inherits' | 'calls'
  PRIMARY KEY (src, dst, kind)
);
CREATE INDEX edges_dst ON edges(dst);
CREATE INDEX edges_src ON edges(src);
```

## indexer.py — what it does

1. Walk all `.py` files under project root, excluding `.venv`, `test_files`, `__pycache__`, `build`, `dist`, `.codegraph`
2. For each file, `ast.parse()` → extract:
   - `ClassDef` → class node; bases → `inherits` edges
   - `FunctionDef` / `AsyncFunctionDef` → function/method node with `ast.get_docstring()` and reconstructed signature
   - `Import` / `ImportFrom` → `imports` edges (file-to-file for internal imports)
   - `Call` nodes inside function bodies → `calls` edges where callee is resolvable
3. Write all nodes to `nodes` table, populate FTS, write all edges
4. After DB is built, generate `CODEINDEX.md` by querying the DB (symbol inventory + import graph)

Run: `python .codegraph/indexer.py` (from project root)

---

## query.py — CLI interface for agents

```
python .codegraph/query.py search <query>          # FTS over names + signatures + docstrings
python .codegraph/query.py context <symbol>        # node info + direct callers/callees/imports
python .codegraph/query.py callers <symbol>        # reverse edge traversal
python .codegraph/query.py callees <symbol>        # forward edge traversal
python .codegraph/query.py explore <symbol> [depth]  # BFS to depth N
```

Output is compact JSON so agents can parse it or read it inline without reading any source file.

---

## CODEINDEX.md — static pre-built reference

Generated by `indexer.py` after DB is built. Structure:
1. **Module map**: each `.py` file → classes + functions (with line numbers)
2. **Import graph**: internal `file → [imported files]` adjacency list
3. **Layer map**: View / Controller / Model / ToolManager / src.printing
4. **Usage note**: "Read this file OR run `python .codegraph/query.py search <symbol>`"

Reading this one file is equivalent to grepping all source files.
