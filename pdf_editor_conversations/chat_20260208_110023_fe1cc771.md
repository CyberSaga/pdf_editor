## 順序說明（先算大小還是先轉向）

**實際流程是：先依 rect 排版 → 再旋轉 → 再依「旋轉後的內容」塞進 rect 來決定要不要縮小。**

1. **先排版**：`insert_htmlbox(rect, ...)` 在我們給的 rect 裡排版（例如 rect 寬 36、高 1190 → 窄長條）。
2. **再旋轉**：排好的內容整塊做 270° 旋轉；所以「排版時的高度」會變成「旋轉後在頁上的寬度」。
3. **再算縮放**：引擎會把「旋轉後的內容」塞進**同一個 rect**；若旋轉後比 rect 大，就會縮小（scale_low=0 時）。  
   之前我們給的是「窄×高」(36×1190)，排版用掉約 36×72（2 行），旋轉後約 **72×36**。要把 72 塞進 rect 的寬 36，就得到 scale = 36/72 ≈ **0.5**，所以字級被壓到約一半。

因此：**不是「先轉向再算幾倍行高」**，而是**先依 rect 排版（行高、行數都在這步決定），再旋轉，再用旋轉後的寬高去和 rect 比、決定縮放**。  
我們之前給的 rect 在「旋轉後的寬度」那一維太小，所以字級被壓縮。

---

## 程式上的改動

對**垂直文字 (90/270)**：

- **先算「旋轉後」需要的寬高**：
  - 排版用掉的高度：`layout_height_pt = num_lines * size * 1.5 + size`
  - 排版寬度：`layout_width_pt = redact_rect.width`
  - 旋轉後在頁上：寬 = `layout_height_pt`，高 = `layout_width_pt` → `rotated_w`, `rotated_h`
- **再給 rect**：  
  `used_side = max(redact_rect.width, redact_rect.height, rotated_w, rotated_h, 1)`  
  用 **正方形 rect** `(x0, y0, x0 + used_side, y0 + used_side)`，讓「旋轉後的內容」在寬、高兩邊都塞得進 rect，這樣 scale 就不會被壓到 0.5。

這樣一來：

- 順序是：**先算排版會用掉多少（幾倍行高）→ 再換成旋轉後的寬高 → 再依這個給 rect**，所以是「先算大小（含旋轉後尺寸），再給文字框，避免旋轉後被壓縮」。
- 字級理論上可以維持接近 1（或至少比 0.5 大）；若 `scale_low=1` 仍回傳 -1，再用的 `scale_low=0` 後備也會比之前少縮一點。

**發生過的兩種情況**

1. **空格消失、其餘字保留**  
   - 編輯後字母在，但**空格**不見或壓成零寬（視覺上像「一部分保留、一部分消失」）。  
   - **處理方式**：空格改為 **&#160;**（不換行空格）並包在 `<span class="helv">` 內。  
   - **結果**：已解決，空格能穩定顯示。

2. **字元截斷／錯亂（例如只顯示「so」或變成「somesc」）**  
   - 「some song.」只顯示「so」或變成「somesc」，後半被截斷或與空格/換行混在一起，即部分字保留、部分字消失或錯位。  
   - **原因**：與「框太小／塞不下」有關，引擎在窄框或旋轉下截斷或換行異常。  
   - **處理方式**：透過**擴高 rect** 與 **scale_low=0 fallback** 一併改善，**沒有**再為「部分截斷」做單獨專案修正。  
   - **結果**：多數情況已隨 rect／fallback 減輕；若仍出現部分消失，可再檢查該區塊是否為垂直文字、rect 是否過小、以及 `spare_height < 0` 時是否有執行 fallback。

---

**簡短結論**

- **空格那一部分的「消失」**：已用 &#160; ＋ helv span 解決。  
- **字元那一部分的截斷／消失**：隨 rect 與 fallback 一併改善，未單獨再修；若再發生，優先查該區塊 rect 與 fallback 是否生效。
