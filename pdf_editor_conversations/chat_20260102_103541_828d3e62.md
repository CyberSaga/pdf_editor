PDF（便攜式文件格式）在設計之初是為了「精確顯示」而非「編輯」，因此編輯 PDF 中的文字涉及多項複雜的技術。以下是與 PDF 文字編輯相關的核心技術：

### 1. 內容流解析與重組 (Content Stream Parsing)

PDF 的文字並非像 Word 那樣以段落形式儲存，而是以「內容流」（Content Streams）中的指令組成（如 `BT` 開始文字、`Tj` 顯示文字、`Tm` 設定矩陣）。

* **座標定位：** 編輯技術需要計算字元在頁面上的精確 X-Y 座標。
* **對齊算法：** 當刪除或增加文字時，編輯器必須重新計算後續字元的位移，以維持視覺上的對齊。

### 2. 字體處理技術 (Font Handling)

這是編輯中最常遇到困難的部分：

* **字體嵌入 (Embedding)：** 若 PDF 未嵌入完整字體，編輯時若本地電腦缺乏該字體，將無法新增文字。
* **字型子集 (Subsetting)：** 為了縮減體積，PDF 常只包含文件中用到的字元。若要編輯的文字不在子集中（例如原本只有「你好」，想改成「您好」），則需要更換或重新嵌入字體。
* **編碼映射 (Encoding/ToUnicode)：** 編輯器需透過 `ToUnicode` 表將 PDF 內部的字元碼（Glyph ID）轉換回可編輯的文字（如 UTF-8/Unicode）。

### 4. 版面分析與流式重構 (Layout Analysis & Reflow)

高品質的編輯器（如 Adobe Acrobat 或 ABBYY）會使用人工智慧來分析頁面結構：

* **邏輯結構識別：** 辨識哪些文字屬於標題、哪些是段落或表格。
* **重排技術 (Reflow)：** 讓固定版面的 PDF 暫時具備像 Word 般的流式編輯體驗，當一行字寫滿時會自動換行。

### 5. 增量更新 (Incremental Update)

為了效率與數位簽章的完整性，PDF 編輯技術通常採用「增量更新」。

* **物件修改：** 編輯器不會重新編寫整個檔案，而是在文件末尾附加變更的物件（Objects），並更新交叉引用表（Cross-Reference Table）。

### 6. 座標變換矩陣 (Coordinate Transformation Matrices)

PDF 中的文字可能經過旋轉、縮放或傾斜。編輯技術需要運用線性代數中的矩陣運算，確保編輯後的文字能與原始的視覺效果（角度、大小）保持一致。

這些技術共同運作，才能解決 PDF 「易讀難改」的本質問題。

* **細節：**
* **子集化 (Subsetting)：** 這是 PDF 編輯最常失敗的原因。如果字體是以子集嵌入（如圖示為 `(Embedded Subset)`），檔案內僅包含已出現的字形（Glyphs）。若要新增原始檔案沒有的字元，編輯器必須能調用系統字體或重新嵌入。
* **ToUnicode：** 若遺失 CMap 或 ToUnicode 表，編輯器會看到「亂碼」或「方塊字」，因為它不知道某個編號對應到哪個 Unicode 字元。

### 5. 增量更新 (Incremental Update)

* **查核結果：正確。**
* **細節：** PDF 允許在不更動原始數據的情況下，將新物件附加在檔案末尾。這對於**數位簽章**非常重要，因為修改後的版本仍能檢視原始簽章時的狀態。

* **查核結果：正確。**
* **細節：** PDF 使用平面幾何的 **Affine Transformation（仿射變換）**。
* **數學公式：** 座標轉換通常表示為 3 x 3 的矩陣：

$$\begin{bmatrix} x' \\ y' \\ 1 \end{bmatrix} = \begin{bmatrix} a & c & e \\ b & d & f \\ 0 & 0 & 1 \end{bmatrix} \begin{bmatrix} x \\ y \\ 1 \end{bmatrix}$$

其中 a, b, c, d 控制縮放與旋轉， e,f 控制平移。編輯器必須處理這套矩陣運算，否則編輯後的文字會位移。

- 若要讓使用者在介面開關：可綁定 `model.use_incremental_save`（例如設定頁的勾選框），無需再改 model 邏輯。

### ROLE_1

為何若有浮水印則不套用增量更新？


### ROLE_2

原因在於 **PyMuPDF 增量更新的前提**，以及「有浮水印時我們是怎麼存檔的」。

## 為何有浮水印就不套用增量更新？

### 1. 增量更新的前提

PyMuPDF 的 `save(path, incremental=True)` 規定：

- **outfile 必須等於 `doc.name`**（存檔路徑 = 這份 doc 當初「從哪個檔案開啟」的路徑）。
- 也就是：**同一份已從磁碟開啟的 doc，存回同一個檔案**，才會在該檔案末尾追加變更（增量更新）。

若 doc 不是從該檔案開啟的（例如在記憶體裡新建、或從別的檔開啟），就不符合這個前提，不能對「那個路徑」做增量更新。

### 2. 有浮水印時的存檔方式

目前程式是這樣做的：

```python
if self.watermark_list:
    tmp_doc = fitz.open()                    # 在記憶體中「新建」一份空白 doc
    tmp_doc.insert_pdf(self.doc)             # 把目前 doc 貼進去
    # ... 在 tmp_doc 的頁面上畫浮水印 ...
    tmp_doc.save(new_path, garbage=0)        # 把「這份新 doc」寫到 new_path
    tmp_doc.close()
```

也就是說：

- 寫入磁碟的是 **`tmp_doc`**，不是 **`self.doc`**。
- `tmp_doc` 是 **在記憶體裡新建** 的（`fitz.open()` 無路徑），沒有「從某個既有檔案開啟」的 `doc.name`。
- 因此對 PyMuPDF 來說，這是一次「把一個新組裝的 doc 寫成一個（可能已存在的）檔案」，而不是「把從該檔案開啟的 doc 存回同一檔案」。
- 在這種情況下，**不符合** `outfile == doc.name`，所以 **不能** 對 `tmp_doc` 使用 `incremental=True` 存到 `new_path`。

所以實作上「若有浮水印則不套用增量更新」——不是刻意排除，而是：**有浮水印時我們本來就是在做「新文檔寫出」，根本不符合增量更新的使用條件**。

### 3. 若堅持「有浮水印也要增量」會怎樣？

若要讓「有浮水印時也走增量」，就必須改成：

- 不要用 `tmp_doc` 寫出整份新 PDF；
- 改為：**直接對 `self.doc`（從原檔開啟的那份）加上浮水印**，然後對 `self.doc` 做 `save(原檔路徑, incremental=True)`。

這樣一來：

- 存檔的 doc 仍是「從原檔開啟」的 `self.doc`，`doc.name == 原檔路徑`，可以符合增量更新前提。
- 但代價是：**浮水印會真的寫進 `self.doc`**，之後 undo/redo、關閉再開，都會帶著浮水印，因為浮水印不再是「僅在儲存時才畫上去」的暫存，而是寫入文件本身。這會改變目前「浮水印只存在於記憶體、直到儲存才一併寫出」的設計。

所以目前設計是：**有浮水印時，寧可做一次完整寫出（不套用增量更新），以維持「浮水印只在存檔時才寫入、不污染 undo 與原 doc」的行為**。

---

**一句話**：有浮水印時我們是「新建一份 doc（原內容 + 浮水印）再寫出」，這份 doc 沒有「從該檔案開啟」的 `doc.name`，因此 PyMuPDF 的增量更新根本不適用；若要適用，就要改為在「從原檔開啟的 doc」上直接加浮水印並做增量存檔，會改變現有浮水印的語意與 undo 行為。

### ROLE_2

已依你的狀況做了兩點修正，摘要如下。

## 1. 增量更新失敗：「incremental needs original file」

- **原因**：PyMuPDF 要求增量儲存時，存檔路徑必須與 `doc.name`（開啟時的路徑）一致，若傳入的 `new_path` 格式不同（例如斜線、大小寫）就會失敗。
- **作法**：在確實要走增量更新時，改為用 `self.doc.name` 當存檔目標，不再用 `new_path`：
  - `save_target = self.doc.name if self.doc.name else new_path`
  - `self.doc.save(save_target, incremental=True)`

## 2. 完整儲存時「cannot remove file ... Permission denied」

- **原因**：在 Windows 上，若對「目前正被自己程式開啟的檔案」做完整覆寫，PyMuPDF 會先試圖刪除/取代原檔，可能觸發「檔案仍被開啟」而 Permission denied。
- **作法**：新增 `_full_save_to_path(path)`，當「要存的路徑 = 目前開啟的檔案」時：
  1. 先寫到暫存檔：`self.doc.save(str(temp_save), garbage=0)`
  2. 關閉目前 doc：`self.doc.close()`
  3. 用暫存檔覆蓋原檔：`shutil.copy2(temp_save, path)`
  4. 刪除暫存檔後，重新開啟原檔：`self.doc = fitz.open(path)`

這樣就不會在檔案仍開啟的狀態下覆寫同一路徑，避免 Permission denied。

## 3. 有浮水印且存回原檔

- 有浮水印時會用 `tmp_doc` 寫出；若寫出目標就是目前開啟的檔案，同樣會遇到「覆寫已開啟檔案」的問題。
- 因此當「有浮水印且 new_path == 目前開啟的檔案」時，也改為：先寫到暫存檔 → 關閉 `self.doc` → 用暫存檔覆蓋 `new_path` → 再 `self.doc = fitz.open(new_path)`。

- 增量更新：改為用 `doc.name` 當存檔路徑，減少「incremental needs original file」。  
- 完整儲存／有浮水印存回原檔：一律改為「先寫暫存 → 關閉 doc → 覆蓋原檔 → 再開啟」，避免 Windows 上「覆寫已開啟檔案」的權限錯誤。
